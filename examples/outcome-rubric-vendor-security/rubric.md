# Rubric: evaluate whether a vendor proposal satisfies our security requirements

Attach as the rubric for an outcome-driven agent; a grader marks each criterion pass/fail with cited evidence.

## Completeness
- [c-s1] The deliverable adequately addresses: From the organization's documented security-requirements source, extract and enumerate each requirement as a discrete, individually-identifiable item. Assign a stable identifier to each, capture its exact criterion text, and label it mandatory or optional. Freeze this list as the fixed reference set used for the rest of the evaluation.
  Evidence required: PASS only if the deliverable contains an enumerated reference list of requirements where each item has: (1) a stable identifier (e.g., REQ-01, SR-1); (2) the exact/verbatim criterion text of the requirement; and (3) a mandatory-or-optional label. The list must be drawn from the organization's documented security-requirements source and presented as the fixed set used for the rest of the evaluation. FAIL if requirements are paraphrased loosely with no identifiers, if mandatory/optional classification is missing, if items are not discrete (multiple requirements merged into one line), or if the source of the requirements is not the organization's own documented requirements.
- [c-s2] The deliverable adequately addresses: For each enumerated requirement, scan the vendor proposal and collect the specific passages or artifacts (with locators/quotes) that explicitly address it. Record the proposal's stated claim verbatim, taking those claims at face value; where the proposal does not explicitly and verifiably address a requirement, record 'no supporting passage found' rather than inferring coverage.
  Evidence required: Scope: this criterion grades the EVIDENCE-COLLECTION step only — the raw record of what the proposal says (or does not say) per requirement. It does NOT grade the verdict assigned to each requirement (that is c-s3). PASS only if, for each enumerated requirement from c-s1, the deliverable records the proposal evidence: specific passages/artifacts with locators or verbatim quotes capturing the proposal's stated claim at face value, OR an explicit 'no supporting passage found'. FAIL if evidence-gathering is done for only some requirements, if proposal claims are editorialized/rebutted instead of recorded verbatim at this step, if quotes lack locators, or if requirements with no coverage are left blank rather than marked 'no supporting passage found'.
- [c-s3] The deliverable adequately addresses: Assign each requirement its own verdict of met / not-met / indeterminate. Mark 'met' only when a cited passage from the previous step supports it, attaching that citation; mark requirements with no explicit, verifiable coverage as not-met or indeterminate. Produce the complete per-requirement breakdown with one verdict per item.
  Evidence required: This criterion is the SINGLE home for verdict-value discipline, the 'met'-must-cite rule, and the 'unaddressed → not-met/indeterminate, never inferred' rule. PASS only if a complete per-requirement breakdown exists with exactly one verdict drawn from {met, not-met, indeterminate} per requirement, where: (a) every 'met' has a specific citation attached (a quote, section/page/line locator, or artifact name from the proposal that a reader could look up); and (b) any requirement whose c-s2 record is 'no supporting passage found' — or that otherwise lacks explicit, verifiable coverage — is marked not-met or indeterminate, never 'met' by inference, assumption, benefit-of-the-doubt, or vendor reputation. Grader verifies a one-to-one mapping by counting: the number of verdicts must equal the number of enumerated requirements from c-s1, each appearing exactly once. FAIL if any requirement is missing, has multiple or no verdict, uses a label outside the three-value set, is marked 'met' without an attached proposal citation (or with only a vague 'the proposal' reference or a citation to the requirement document instead of the proposal), or is marked 'met' despite lacking explicit verifiable coverage.
- [c-s4] The deliverable adequately addresses: Reduce the per-requirement breakdown into an overall verdict: 'satisfies' only if every mandatory requirement is met; if any single mandatory requirement is not-met or indeterminate, conclude 'does not satisfy'. Present the overall verdict together with the full per-requirement breakdown, never as a standalone aggregate.
  Evidence required: This criterion is the SINGLE home for the aggregation logic and the 'no standalone aggregate without breakdown' rule. PASS only if the deliverable presents an overall verdict computed by the stated rule: 'satisfies' iff every mandatory requirement is met; 'does not satisfy' if any single mandatory requirement is not-met or indeterminate. The overall verdict must appear together with the full per-requirement breakdown, and the grader must be able to confirm the overall conclusion is consistent with the individual verdicts (check the mandatory items). FAIL if the overall verdict contradicts the breakdown (e.g., 'satisfies' while a mandatory item is not-met/indeterminate), if it is presented as a standalone aggregate without the accompanying breakdown, or if the mandatory-only aggregation logic is not applied.

## General
- [c-responsive] The deliverable directly addresses the stated objective — not a generic treatment near the topic.
  Evidence required: PASS only if the deliverable evaluates THIS specific vendor proposal against THIS organization's security requirements — not a generic essay on vendor security or how-to-evaluate-vendors advice. Grader must find: (1) references to concrete, named requirements drawn from the organization's own security-requirements source; (2) references to concrete content from the specific vendor proposal (vendor name, product, or quoted proposal text); (3) an actual assessment/verdict, not just background. FAIL if the text could be copy-pasted for any other vendor or any other requirement set, if it discusses security topics generally without producing a proposal-vs-requirements assessment, or if it substitutes advice ('you should verify encryption') for a finding.
- [c-grounded] Claims in the deliverable are grounded in provided material or clearly flagged as uncertain; nothing is fabricated.
  Evidence required: Scope: this criterion targets fabrication and honesty about uncertainty ONLY — it does NOT re-check the per-verdict citation discipline for 'met' labels (that is owned exclusively by c-s3) nor the treatment of unaddressed requirements (owned by c-s2/c-s3). PASS only if every substantive claim about what the proposal says is either traceable to provided material (quoted or locator-referenced proposal passages, or named requirement text) OR explicitly flagged as uncertain (e.g., 'proposal does not state', 'indeterminate — ambiguous wording'). FAIL if the grader finds any fabricated capability, invented proposal quote, or a factual assertion attributed to the proposal that has no basis in the provided material. Spot-check: pick 2-3 asserted facts about the proposal — each must be backed by a citation or an explicit uncertainty flag. Any invented detail = FAIL.
