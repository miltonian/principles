# MCP for Builders: A Practical, Hands-On Primer

Model Context Protocol (MCP) is the connective tissue that lets an AI agent reach out of its chat window and actually *do* things — read your files, query your database, open a pull request, search the web — through one standardized interface instead of a dozen bespoke ones. This primer is written for a developer with a Python or TypeScript background who wants to build with it: what MCP actually is, why 2025 turned it into infrastructure, how it differs from raw APIs and framework tools, a copy-pasteable path to your first working server, the servers and host applications you'll actually use, the disciplines that make servers reliable and safe, and an honest read on how far this can go — including whether it can automate whole jobs.

## What MCP Actually Is

The **Model Context Protocol** is an open-source, open standard defining a **client–host–server** protocol that standardizes how an AI application connects to external systems to *discover* and *invoke* capabilities. The official docs call it "an open-source standard for connecting AI applications to external systems" and liken it to "a USB-C port for AI applications" — one standardized connector replacing bespoke integrations. Under the hood it is built on **JSON-RPC 2.0** as a **stateful session protocol** with explicit **capability negotiation** at connection time.

Three roles stay crisply distinct throughout everything below:

- **Host / Client (the AI-agent side).** The **host** is the AI application that coordinates one or more MCP clients (Claude Desktop, VS Code, Cursor). It holds the LLM, manages consent and security policy, and decides *when* to act. Each **client** maintains a **1:1 connection** to a single server and handles protocol negotiation, capability exchange, and discovery.
- **Server.** "A program that provides context to MCP clients." It exposes capabilities and operates "independently with focused responsibilities." Crucially, a server is a *passive* capability provider — by design it "should not be able to read the whole conversation, nor 'see into' other servers."
- **The three server primitives** (the docs' exact term):
  - **Tools** — executable functions the agent can invoke to perform actions (file operations, API calls, database queries).
  - **Resources** — data sources providing contextual information (file contents, database records, API responses).
  - **Prompts** — reusable templates that structure interactions (system prompts, few-shot examples).

Discovery and invocation use method families like `tools/list`, `resources/list`, and `tools/call`. Clients may also expose their own primitives — **sampling, elicitation, and logging**. Two standard **transports** carry the same JSON-RPC messages: **stdio** (a local subprocess reading and writing over stdin/stdout) and **Streamable HTTP** (HTTP POST plus optional Server-Sent Events for remote servers). Transports are an implementation layer distinct from the protocol's data layer — a useful separation to keep in mind as you scale from a local tool to a hosted service.

## Why MCP Became a Buzz in 2025 — Why Now?

MCP was open-sourced by Anthropic on **November 25, 2024**. Because that release landed at year-end, **2025 was MCP's first full calendar year as a public standard** — and the excitement is exactly what a well-timed standard looks like when it drops into a market starving for one.

**The pain point: the N×M integration explosion.** Before a shared protocol, connecting AI applications to the outside world was a combinatorial mess. With **N** agent applications (a coding assistant, a chatbot, an internal ops agent) and **M** systems to reach (GitHub, Slack, a Postgres database, the local filesystem), every team hand-wrote a bespoke connector for each pair — custom auth, custom schemas, custom call conventions. That's **N×M** brittle integrations: 3 apps × 4 tools = 12 separate glue layers, each maintained, re-debugged, and re-invented by the next team. A shared protocol collapses this to **N+M**: build one GitHub server, one Slack server, one Postgres server, one filesystem server, and *any* compliant host can use all of them — 4 servers + 3 hosts = 7 components instead of 12, and the ratio only improves as the ecosystem grows. The server author integrates once; every agent reuses it. This is the same economics that made shared drivers and USB win over per-device wiring.

**Standardized runtime discovery.** The N+M win only materializes because capabilities are discoverable in a uniform way. A host connects and asks, at runtime, "what can you do?" via `tools/list`, negotiating capabilities at `initialize`, rather than hardcoding tools at build time. Proprietary tool schemas would just move the N×M problem to the framework layer; a neutral wire protocol means a server author targets *one* interface and reaches *every* compliant host. Interoperability, not any single feature, is the product.

**Why 2025 specifically — a flywheel, not a coincidence.** Three forces converged. First, **timing**: open-sourced in late 2024, 2025 was the first year builders could adopt at scale. Second, **agentic LLMs matured**: 2025's models became genuinely good at tool use and multi-step autonomy, making reliable external tool access urgent rather than optional — MCP arrived exactly when agents needed hands. Third, **network effects**: more servers make adopting a host more valuable, which pulls in more hosts, which incentivizes more servers.

**Real adoption signals — named actors, dated events.** OpenAI adopted MCP in **March 2025** across its Agents SDK, Responses API, and ChatGPT (Sam Altman confirmed on March 26). Google committed in **April 2025**, with Gemini support and later official managed MCP servers on Google Cloud. Official SDKs now span TypeScript and Python as mature Tier-1 languages, with an official Go SDK reaching 1.0 later in the year. Public registries emerged (Smithery, the official MCP Registry). And on **December 9, 2025**, Anthropic donated MCP to the Linux Foundation's **Agentic AI Foundation (AAIF)**, co-founded with Block and OpenAI and backed by Google, Microsoft, AWS, Cloudflare, and Bloomberg — moving MCP from one vendor's project to neutral, multi-vendor stewardship. A standard that solves a real N×M cost, dropped exactly as agents got good enough to need it, adopted within months by every major model vendor and then handed to a neutral foundation — that convergence, in MCP's first full year, is the buzz.

## MCP vs. Raw APIs vs. Framework Agent Tools

Three things are easy to blur: the **AI agent** (the LLM-driven client/host that decides *when* to act), the **MCP server** (a process exposing tools, resources, and prompts over the protocol), and an **API** (any arbitrary remote interface — REST/GraphQL/RPC — with its own bespoke schema and auth). MCP does not replace APIs; it standardizes the *agent-facing* layer that sits on top of them. The real contrast is between three integration strategies for getting a capability into an agent.

| Axis | Raw API integration | Framework agent tool (LangChain/LlamaIndex-style) | MCP server |
|---|---|---|---|
| **Interface contract** | Bespoke per API — glue for auth, schema, pagination, error shapes each time | The framework's tool abstraction and types | Uniform, self-describing contract over JSON-RPC with capability negotiation |
| **Discovery** | None — endpoints hardcoded from docs | Registered at build time | Runtime enumeration (`tools/list`, `resources/list`) with schemas |
| **Portability across hosts** | Coupled to your custom wrapper code | Coupled to one framework's runtime | Any MCP-compatible host can use the same server |
| **Coupling** | Tight to the service and your code | Tight to the framework | Loose — the protocol boundary isolates agent from server |
| **Best fit** | Deterministic backend-to-backend, latency-critical, no LLM in the loop | Single-framework apps, rapid prototyping, in-process logic | Reusable, discoverable, host-agnostic capabilities across many agents |

**What MCP standardizes that a raw API does not.** With a raw API, every integration is bespoke: you hand-write auth flows, translate schemas, handle pagination, and normalize error shapes per service. MCP replaces that per-integration glue with one contract — a common transport and handshake plus self-describing tools, resources, and prompts — so an agent talks to *any* compliant server the same way. The underlying API doesn't vanish; the MCP server usually wraps it. What changes is that the agent-facing surface becomes uniform instead of one-off.

**Why runtime discovery beats hardcoded integrations.** Framework tools and hand-rolled wrappers are registered at build time — adding or upgrading a capability means editing code and redeploying. An MCP host instead queries the server at runtime (`client → server: tools/list`; `server → client: [{name, description, inputSchema}, ...]`), so you can add a new server or upgrade an existing one and a running agent picks up the change without recompilation. **Portability** is the second differentiator: like the Language Server Protocol lets one language server serve any compliant editor, one MCP server can serve many agents — Claude, an IDE, a custom agent — without rewrites.

**Where each still fits — a fair read.** None is universally correct. Raw APIs remain the substrate: direct calls are right for deterministic backend-to-backend flows, latency-critical paths, or any flow with no LLM in the loop — a protocol layer there buys nothing. Framework tools stay convenient for tightly-coupled, single-framework apps and rapid prototyping. MCP shines when the *same* capability should be consumable by many agents, where the protocol boundary is repaid by not re-wiring each integration per host. The decision is about reuse and portability, not raw capability.

## Getting Started: Build and Run Your First MCP Server

You'll build a local server that the host launches as a subprocess and talks to over **stdio** (JSON-RPC on stdin/stdout). Target the current *stable* SDKs — Python `mcp` and TypeScript `@modelcontextprotocol/sdk`. A v2 pre-release with renamed symbols exists on the SDKs' `main` branches; avoid it for a first build.

### 1. Prerequisites

- **Python track:** Python **3.10+** and the [`uv`](https://docs.astral.sh/uv/) package manager (or `pip`). Install with CLI extras: `uv add "mcp[cli]"` or `pip install "mcp[cli]"`.
- **TypeScript track:** **Node.js LTS (v18+)** plus `npm`/`pnpm`. Install with `npm install @modelcontextprotocol/sdk zod`.
- **Mental model:** you're writing a small program that speaks JSON-RPC over stdout. The host spawns it, calls `tools/list` to discover your tool, and `tools/call` to run it.

### 2. Choose an SDK

Pick the language your surrounding stack already uses: **Python** if you'll wrap data/ML/pandas or Python APIs; **TypeScript** if you live in Node/web tooling or want an npx-installable server. Both are first-party and feature-equal for this build.

### 3. Hello-world server (one tool)

**Python** — create the project and `server.py`:

```bash
uv init mcp-add-server && cd mcp-add-server
uv add "mcp[cli]"
```

```python
from mcp.server.fastmcp import FastMCP

# The server name is what the host displays.
mcp = FastMCP("add-server")

@mcp.tool()
def add(a: int, b: int) -> int:
    """Add two numbers and return the sum."""
    return a + b

if __name__ == "__main__":
    # Runs the JSON-RPC loop over stdio (stdin/stdout).
    mcp.run(transport="stdio")
```

The `@mcp.tool()` decorator registers `add`, derives its input schema from the type hints, and uses the docstring as the tool description.

**TypeScript** — after `npm install @modelcontextprotocol/sdk zod`, create `src/server.ts`:

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "add-server", version: "1.0.0" });

server.registerTool(
  "add",
  {
    title: "Add",
    description: "Add two numbers and return the sum",
    inputSchema: { a: z.number(), b: z.number() }, // raw Zod shape, not z.object(...)
  },
  async ({ a, b }) => ({ content: [{ type: "text", text: String(a + b) }] }),
);

await server.connect(new StdioServerTransport());
```

Note `inputSchema` is a Zod *raw shape* (`{ a: z.number() }`), not `z.object(...)`. Add `"type": "module"` to `package.json`, then compile with `npx tsc` to emit `build/server.js`.

### 4. Wire it into a host (Claude Desktop)

Open **Claude Desktop → Settings → Developer → Edit Config**, which opens `claude_desktop_config.json` (macOS: `~/Library/Application Support/Claude/`; Windows: `%APPDATA%\Claude\`). Register under `mcpServers` with **absolute paths** — the host has no working directory of yours:

```json
{
  "mcpServers": {
    "add-server": {
      "command": "uv",
      "args": ["--directory", "/ABSOLUTE/PATH/TO/mcp-add-server", "run", "server.py"]
    }
  }
}
```

For the TypeScript build use `"command": "node"` with `build/server.js`. The host spawns this command as a subprocess and speaks MCP over its stdio.

### 5. Connect and test

Fast dev loop — the **MCP Inspector**, the official test client, runs via `npx` with no install:

```bash
npx @modelcontextprotocol/inspector uv --directory /ABSOLUTE/PATH/TO/mcp-add-server run server.py
```

Open the Inspector URL, go to the **Tools** tab, click **add**, enter `a` and `b`, and confirm the result. Then end-to-end: fully restart Claude Desktop, click the MCP indicator to confirm `add` is listed, and ask *"Use the add tool to add 21 and 21"* — expect `42`.

**Troubleshooting.** Server not appearing → validate the JSON (a trailing comma breaks it) and fully restart; check logs (macOS `~/Library/Logs/Claude/mcp*.log`). Path issues → `command`/`args` must be absolute. **stdout pollution (the classic gotcha)** → stdio uses stdout for JSON-RPC, so a stray `print()`/`console.log()` corrupts the channel; send all logs to stderr (Python `logging` defaults there; use `console.error` in Node). Interpreter mistakes → ensure `uv`/`node` are on the host's PATH; run the exact command in a terminal first. From here, add **resources** and **prompts** using the same registration pattern.

## Common MCP Hosts and Applications That Use These Servers

A **host** embeds an MCP client, holds the LLM, and decides *when* to invoke a server's capabilities — "host" ≠ "server." This landscape moves fast: support is added, expanded, and occasionally removed rapidly, so re-verify against each product's docs. Capability coverage (which transports, which primitives) is often version-gated.

**Desktop / web AI assistants**
- **Claude Desktop & Claude.ai (Anthropic)** — the reference host. Desktop connects to **local stdio servers** via a JSON config; Claude.ai connects to **remote servers** through OAuth "Custom Connectors" (enter a URL, complete OAuth, pick tools/resources/prompts). It has the fullest primitive coverage — **tools, resources, prompts, and sampling** — plus per-tool permission controls.
- **ChatGPT (OpenAI)** — connects to remote MCP servers via hosted connectors, confirming the protocol is genuinely cross-vendor. Usable without local setup.
- **LibreChat (open-source)** — a self-hostable, multi-provider chat UI that attaches MCP servers as **tools** inside its customizable agents, regardless of the backing model — one MCP config works across LLMs.

**AI-native IDEs / coding agents (agentic tool-calling loops)**
- **Claude Code (Anthropic)** — a terminal/CLI coding agent; `claude mcp add` supports **stdio, SSE, and Streamable HTTP** and local/project/user scopes, surfaces server **prompts as slash commands** and **resources** as referenceable context, and plugs servers straight into its autonomous multi-step loop (read an issue tracker, query a DB, open a PR).
- **Cursor (Anysphere)** — an early, mature host configured via `.cursor/mcp.json` (project) or `~/.cursor/mcp.json` (global); supports **tools, resources, prompts, and roots**, with one-click marketplace install and approval-gated Agent/Plan mode.
- **VS Code + GitHub Copilot (Microsoft/GitHub)** — agent mode consumes **tools, resources, and prompts** (invoked as `/server.prompt`) via `.vscode/mcp.json`, over stdio and HTTP, with a built-in **server gallery** and workspace-shareable config.
- **Cline (open-source VS Code extension)** — an autonomous agent that lists each server's **tools and resources**, executes with step-by-step approval, and can even help you *create* new servers from natural-language descriptions.
- **Windsurf Editor (Codeium)** — an agentic IDE ("AI Flow") that natively consumes MCP **tools** via an `mcp_config.json`.
- **Zed (Zed Industries)** — a fast native editor exposing server **prompts as slash commands** with tool integration (does **not** support resources, per the official matrix).
- **Continue (open-source)** — a code assistant with **full support for all MCP features** across VS Code and JetBrains.
- **Goose (Block, open-source)** — an extensible agent treating MCP servers as first-class installable extensions.
- **Warp (Warp.dev)** — an agentic terminal that configures MCP servers to extend its command-line agents.

If you're choosing where to build against first, the coding hosts (Claude Code, Cursor, VS Code + Copilot, Cline) give you the richest agentic loops with approval gating; the desktop assistants (Claude Desktop, ChatGPT) are the easiest place to demo a local stdio server.

## Development Best Practices for Building MCP Servers

The goal is servers an LLM can use *reliably and safely*. A few disciplines matter most.

**Clear tool schemas and descriptions.** The tool name, description, and input JSON Schema are the model's *only* interface to your tool — write them for an LLM reader. Prefer `search_orders(customer_id: str, status: enum[open,shipped,cancelled])` with a purpose description over a vague `orders(query)`. Use unambiguous, action-oriented names, well-typed and constrained parameters, and a note on when *not* to use the tool. Keep the surface area small so the model can choose reliably.

**Transports.** The spec defines exactly two standard transports: **stdio** (local subprocess, simplest, no network exposure — ideal for local dev tools and desktop hosts) and **Streamable HTTP** (for hosted/multi-user servers). The current reality worth internalizing: the older standalone HTTP+SSE transport is **deprecated and superseded** by Streamable HTTP — SSE now exists only as optional streaming *inside* it. Rule of thumb: stdio for local/desktop, Streamable HTTP for remote/shared.

**Scoping and least-privilege.** Expose the minimum set of tools and resources, narrow parameter ranges, separate read-only from write, and avoid broad "do anything" tools. This helps both reliability (the model picks the right tool) and safety (smaller blast radius).

**Error handling.** Return structured, model-readable errors so the agent can recover or retry — don't crash the transport. Distinguish protocol-level errors from tool-execution errors, validate inputs *before* side effects, set timeouts, and make retried actions idempotent.

**Security and auth.** A server can expose sensitive systems, so mind the trust boundary. For remote (HTTP) servers, current guidance is **OAuth 2.1-based** authorization with **audience binding** (RFC 8707) and **no token passthrough** (to avoid confused-deputy attacks); stdio servers use environment credentials rather than OAuth. Never hard-code secrets. Sanitize inputs, and treat content a tool *returns* as untrusted — indirect prompt injection via tool output is a real risk. Require human consent for destructive actions.

**Testing and the Inspector.** Use the MCP Inspector (`npx @modelcontextprotocol/inspector node build/index.js`) to manually exercise tools, resources, and prompts before wiring to a host; add unit tests against your tool handlers directly; and run the server against a real host as an integration check.

**Versioning.** MCP negotiates a protocol version at `initialize`. Semantically version your own server and treat schema changes with discipline — additive changes are safe; breaking changes need deprecation, not silent removal.

## Where MCP Delivers the Most Value: A Grounded Domain Assessment

Reasoning from the servers and hosts above, MCP pays off wherever a workflow chains reusable, discoverable capabilities. Each domain below is justified by naming specific servers and the hosts where the workflow realistically runs.

**1. Software development.** *Servers:* GitHub, GitLab, Git, Filesystem, Sentry, Fetch, Sequential Thinking. *Hosts:* Claude Code, Cursor, VS Code + Copilot, Cline, Windsurf, Zed, Continue. *End-to-end:* an agent reads a live crash from **Sentry** → locates the offending code via **Git/Filesystem** → drafts a fix → opens a PR through **GitHub** — all inside Claude Code's or Cursor's agent loop. This is the highest-maturity domain by evidence; read/triage largely automates, but the human gates the **write side-effect** (merging), because the agent can't verify semantic correctness and write scopes require human-held tokens.

**2. Data & analytics access.** *Servers:* PostgreSQL, SQLite, Supabase. *Hosts:* Claude Desktop, Cursor, Claude Code. *End-to-end:* query a **Postgres** table → summarize → post the digest to **Slack**. The read-path is strong, but note these DB servers are typically **read-only** — a deliberate least-privilege choice. Human judgment remains for query ambiguity (which table, which definition of a metric).

**3. Knowledge management.** *Servers:* Notion, Google Drive, Atlassian (Confluence), Memory, Fetch. *Hosts:* Claude Desktop, Claude.ai, VS Code + Copilot. *End-to-end:* search **Confluence** and **Drive** → synthesize → write a new **Notion** page, persisting durable facts via **Memory**. Retrieval and summarization automate well; **writing** needs review, and each source needs its own auth/consent.

**4. DevOps / automation.** *Servers:* Cloudflare, Sentry, GitHub, Filesystem, Fetch. *Hosts:* Claude Code, Warp, Cursor. *End-to-end:* detect a **Sentry** error spike → inspect **Cloudflare** Workers/KV/D1 state → open a remediation PR via **GitHub**. Diagnosis automates; **production side-effects** (deploys) are the hard human gate — irreversible actions with privileged credentials.

**5. Web research & browser automation.** *Servers:* Brave Search, Playwright, Puppeteer, Fetch. *Hosts:* Claude Desktop, Cursor, Claude Code. *End-to-end:* **Brave Search** for candidates → **Playwright** to navigate/extract → **Fetch** to clean content → summarize. Collection automates well; verification and untrusted-content prompt-injection keep a human in the loop before acting on scraped data.

**6. Personal productivity.** *Servers:* Notion, Slack, Google Drive, Google Maps, Brave Search, Time, Fetch. *Hosts:* Claude Desktop, ChatGPT. *End-to-end:* research with **Brave Search** → compile into **Notion** → pull travel directions via **Google Maps**. Convenient but assistive — the workload per task is small, so "automation" is help, not replacement.

**7. Customer support (evidence-thin — flagged).** *Servers:* Slack, Atlassian (Jira), Stripe, Sentry — but **no dedicated help-desk server** (Zendesk/Intercom) on the mid-2025 list. *Hosts:* Claude Desktop, LibreChat. *Assembled example:* read a **Slack** thread → check billing via **Stripe** → look up a linked bug in **Jira** → draft a reply. The pattern fits, but the canonical support-desk tools are absent — a plausible-but-underserved domain, not a mature one. Stripe write actions (refunds) are high-consequence, and support content is untrusted user text, so injection risk is acute.

|---|---|---|---|
| Software development | GitHub, Git, Sentry, Filesystem | Claude Code, Cursor, VS Code | Sentry error → Git fix → GitHub PR |
| Data & analytics | PostgreSQL, SQLite, Supabase | Claude Desktop, Cursor | Postgres query → summary → Slack |
| Knowledge management | Notion, Drive, Confluence, Memory | Claude.ai, Claude Desktop | Confluence+Drive → Notion page |
| DevOps / automation | Cloudflare, Sentry, GitHub | Claude Code, Warp | Sentry spike → Cloudflare → PR |
| Web / browser automation | Brave Search, Playwright, Puppeteer | Claude Desktop, Cursor | Brave → Playwright → summarize |
| Personal productivity | Notion, Slack, Drive, Maps | Claude Desktop, ChatGPT | Brave → Notion → Maps |
| Customer support (thin) | Slack, Stripe, Jira, Sentry | Claude Desktop, LibreChat | Slack thread → Stripe → Jira → reply |

Across every domain the same three limits recur — human-held **auth/OAuth**, gated **write side-effects**, and **verification of untrusted output** — which are the concrete reasons no domain here is fully hands-off.

## Can MCP Automate Entire Human Workloads?

The question people really want answered is "will this replace jobs?" — and it's usually mis-scoped. A job is not one thing; it's a *bundle* of tasks plus judgment, accountability, context-switching, stakeholder handling, and ownership of edge cases. Automation attacks tasks first. MCP is a lever on tasks specifically: as a standardized protocol, it widens the set of tools and data an agent can reach and act on. That is an *integration* gain — more reachable, composable actions — not a gain in reasoning or autonomy.

**What MCP-connected agents demonstrably do today (task level), human-supervised:** in software dev, read a crash report, locate the code, draft and open a PR for review; in data/analytics, run a read-only query and summarize or chart it; in knowledge management, search internal docs and synthesize a draft; in DevOps, detect a spike, inspect state, propose a remediation; on the web, search, navigate, extract, and summarize; in support/productivity, triage a thread, pull related context, draft a reply. Each is a discrete, bounded slice — exactly what an agent-plus-tools loop handles well.

**What is not demonstrated: owning a whole role.** The barriers are concrete. **Reliability / error compounding** — chaining many tool calls multiplies failure probability, and long autonomous chains drift. **Accountability** — someone must sign off on consequential actions; a protocol confers no liability. **Judgment / edge cases** — ambiguous metrics, "which table," canonical accuracy are human calls. **Security / blast radius** — write side-effects (merges, deploys, refunds) stay gated behind human-held OAuth/tokens, and untrusted tool output carries prompt-injection risk. **Out-of-system context** — much of a role lives in tacit priorities, hallway decisions, and relationships no server exposes. And **access ≠ correct action** — MCP standardizes what an agent can *reach*; it guarantees nothing about whether the action is *right*.

**Realistic near-term shape:** augmentation, not elimination. Agents take task-level load off humans; specific *workflow slices* get automated end-to-end within guardrails (approval gates, least-privilege scopes, idempotent retries); roles get *reshaped* as routine work is offloaded and the human concentrates on judgment, exceptions, and sign-off. Human-in-the-loop is the design center, not a temporary crutch. Bottom line: MCP is an integration/plumbing layer that expands what an agent can reach and makes those integrations reusable across hosts. It does not, by itself, confer the reliability, judgment, or accountability required to own a full human role. Build for augmented workflows with a human in the loop — and treat "it replaces the job" as an unproven claim, not a roadmap.

## Outlook: How Prominent Is MCP Likely to Become?

*Everything here is forecast and scenario. The signals are real as of this snapshot; the inferences are probabilistic bets, to be read as "if current trends hold," not certainty.*

**Where things stand.** Governance has moved off a single vendor — Anthropic donated MCP to the Linux Foundation's Agentic AI Foundation (December 2025), with AWS, Anthropic, Block, Bloomberg, Cloudflare, Google, Microsoft, and OpenAI among platinum backers. Adoption is large and cross-vendor, with first-class client support across ChatGPT, Claude, Cursor, Gemini, Microsoft Copilot, and VS Code, and a published 2026 roadmap prioritizing transport evolution, agent-communication lifecycles, governance maturation, and enterprise readiness. (Circulating download and server counts are vendor-reported, not independently audited.)

**Bull case.** If these signals compound, MCP appears positioned to settle into the role a shared protocol plays after neutral stewardship arrives — the foundation itself invokes Kubernetes and PyTorch as analogies. Because MCP standardizes tool *discovery* and collapses the N×M problem, the flywheel should keep turning. "Does it speak MCP?" could plausibly become a default procurement question for AI tooling by late 2026–2027, the way REST or OpenAPI became table stakes. Credible, but not assured.

**Bear case.** Security is the sharpest downside: tool poisoning and indirect prompt injection are documented, not hypothetical, and a high-profile breach could stall enterprise adoption pending hardening. A cost/efficiency backlash is visible too — tool definitions can consume a large share of the context window, and some teams report high token overhead versus CLI equivalents, feeding an "is MCP dead?" narrative. Fragmentation and spec churn are risks as major vendors propose diverging extensions, and complementary protocols like Google's Agent-to-Agent (A2A, framed as agent-to-agent vs. MCP's agent-to-tool) could blur into rivalry.

**Most-likely trajectory.** A reasonable base case: MCP becomes a durable *de facto* standard for agent-to-tool integration over the next one to two years — entrenched rather than universal, coexisting with A2A-style protocols and proprietary escape hatches for latency- or token-sensitive paths. Confidence: moderate. The swing variables to watch: whether the foundation ships enforceable security/auth and verified-registry standards; whether token-efficiency improves enough to blunt the cost critique; whether extension proposals converge or fork; and whether a major vendor defects. Progress on the first three pushes toward the bull case; a defection or a marquee security incident pushes toward the bear case. For you as a builder, the practical read is simple: the protocol is stable enough and adopted widely enough that building against it today is a safe bet — design for least-privilege and human-in-the-loop, and you'll be resilient to whichever scenario unfolds.
