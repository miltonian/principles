# MCP Servers and AI Agents: A Builder's Guide (as of 2026-07-04)

## Anchor definitions

The **Model Context Protocol (MCP)** is an open client–server *specification* — a standard, not a product, a hosted service, or a single server you install. It standardizes how LLM applications connect to external capabilities by defining exactly three primitive kinds a server can expose: **tools** (callable actions/functions the model can invoke), **resources** (readable data/context the model can pull in), and **prompts** (reusable prompt templates). These primitives are exchanged over a *defined transport* — a specified wire format between client and server — so that any spec-compliant client can talk to any spec-compliant server. That interoperability is the entire point of a protocol. Put plainly: MCP is the rulebook, not a running program you download.

An **AI agent** is an LLM-driven system that autonomously selects and invokes external tools in a loop (observe → decide → act → observe again) until its goal is met. An **MCP server** is the tool/data provider; an **MCP client** is the side that connects to and consumes those capabilities. The agent — or the app hosting it — *is* the MCP client. The direction is unambiguous: the server serves, the client consumes.

## Why MCP became a buzz in 2025

Looking back from mid-2026, the 2025 excitement is easy to explain — and easy to verify. The buzz was not that a company shipped a product; it was that a *protocol* crossed the threshold from proposal to de facto standard. The inflection was Q1–Q2 2025, when competitors — not just Anthropic — began shipping support.

The structural reason a protocol spreads is that it dissolves an integration explosion. Before a standard, connecting *M* agent applications to *N* tools is an M×N problem: every app hand-builds a bespoke connector to every tool. Because MCP standardizes the interface, one compliant server works across every compliant client, collapsing the cost to M+N — build a server once, reach every agent; build a client once, reach every server. That is precisely the network-effect dynamic that produces steep adoption curves: each new server makes every client more valuable, and vice versa.

That mechanism is why the vendor commitments mattered far more than any press release. When OpenAI, Google, Microsoft, and AWS each committed on dated timelines, they weren't endorsing a company — they were agreeing to speak a shared language, which is what turns an M×N mess into an M+N ecosystem. The registry passing ~2,000 curated entries and the servers repo passing ~88k stars are the visible surface of that same flywheel: the more servers exist, the more reason there is to build a client, and the more clients exist, the more reason there is to build a server. The buzz, in short, was the sound of a coordination problem being solved in public.

## How MCP differs from plain APIs and ad-hoc tool integrations

**The "before MCP" world.** Connecting an agent to an external capability meant wrapping a REST or GraphQL API in a hand-written, framework-specific tool adapter: author a function-calling schema the model could understand, wire up that API's particular auth (API keys, OAuth flows, header conventions), and write bespoke code to parse each endpoint's responses back into something the agent loop could consume. That glue was rarely portable — an adapter written for one framework's tool interface generally did not work in another. Every *(framework, API)* pairing was its own small integration project.

**The combinatorial problem.** With **M** agent clients each needing custom code to talk to **N** tools, you build and maintain **M × N** integrations. *Illustratively:* 5 agent applications each integrating 20 tools requires 5 × 20 = **100** separate integrations — and the cost is ongoing, since any API or adapter-contract change ripples across many of them.

**The differentiator: standardization.** MCP defines one shared protocol for tool discovery, invocation, resources, and prompts over a defined transport. Each client implements the protocol **once** to become an MCP client (M implementations); each tool provider implements a server **once** (N implementations). Any compliant client then talks to any compliant server **with no per-pair code**. The total collapses from M × N to **M + N** — reusing the example, **5 + 20 = 25** instead of 100, and the saving widens as the ecosystem grows because new tools are additive (+1), not multiplicative (×M). Think USB-C: once a device speaks the standard, it works with every compliant host without a custom cable each time.

**What MCP does *not* eliminate.** Servers still call the underlying REST/GraphQL APIs internally — MCP is a standard wrapper, not a replacement. Auth, rate limits, and business logic move *behind* the server boundary; they don't vanish. The protocol adds a small, fixed indirection cost in exchange for turning a multiplicative burden into an additive one.

| Dimension | Plain API | Ad-hoc agent tool | MCP server |
|---|---|---|---|
| Who writes the glue | Consumer, per app | App dev, per framework | Written once, in the server |
| Reusable across clients | No | No (framework-specific) | Yes (any compliant client) |
| Capability discovery | Out-of-band docs | Hardcoded per adapter | Introspectable via the protocol |
| Coupling to a framework | N/A | Tight | Decoupled (protocol-mediated) |
| Maintenance as it scales | M × N | M × N | M + N |

## Getting started: build a working MCP server and call it from an agent

By the end of this section you'll have a running MCP server exposing one tool (`add`), verified in the MCP Inspector, then called by a programmatic client that lists and invokes the tool exactly as an agent's tool-use loop does. Recall the roles: the **server** is the tool/data provider, and your agent (or its host) connects to it as the **client**.

### 1. Environment & SDK choice
Default to the **official TypeScript SDK (`@modelcontextprotocol/sdk`) on Node.js 18+**: it is first-party, its types make the three primitives editor-discoverable, and server and client live in one package. The **official Python SDK (`mcp`, `FastMCP` API)** is equally supported — pick it if your tools wrap Python-native libraries. Prerequisites: Node.js 18+ (`node --version`) and npm; a TypeScript-aware editor; *(Python path only)* Python 3.10+.

### 2. Build your first MCP server
```bash
mkdir calc-mcp && cd calc-mcp
npm init -y
npm pkg set type=module
npm install @modelcontextprotocol/sdk zod
npm install -D typescript @types/node
```
`tsconfig.json`:
```json
{
  "compilerOptions": {
    "target": "ES2022", "module": "NodeNext", "moduleResolution": "NodeNext",
    "outDir": "build", "strict": true, "esModuleInterop": true, "skipLibCheck": true
  },
  "include": ["*.ts"]
}
```
`server.ts` (exposes one tool; servers can also expose **resources** via `registerResource` and **prompts** via `registerPrompt` — omitted to stay minimal):
```ts
// server.ts
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "calc-server", version: "1.0.0" });

// inputSchema is a Zod *raw shape*, which the SDK converts to the tool's JSON Schema.
server.registerTool(
  "add",
  {
    title: "Add two numbers",
    description: "Return the sum of a and b",
    inputSchema: { a: z.number(), b: z.number() },
  },
  async ({ a, b }) => ({ content: [{ type: "text", text: String(a + b) }] })
);

async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
  console.error("calc-server running on stdio"); // stdout is reserved for the protocol
}
main().catch((err) => { console.error("Fatal:", err); process.exit(1); });
```
Python equivalent (server only):
```python
# server.py
from mcp.server.fastmcp import FastMCP
mcp = FastMCP("calc-server")

@mcp.tool()
def add(a: int, b: int) -> int:
    """Return the sum of a and b."""
    return a + b

if __name__ == "__main__":
    mcp.run()  # defaults to stdio
```
Build: `npx tsc` → emits `build/server.js`.

### 3. Transports
Two transports to choose between. **stdio** — the client launches the server as a local subprocess; messages flow over stdin/stdout. Zero network setup; ideal for local dev, CLI tools, desktop-host integrations. **Streamable HTTP** — the current remote transport (it superseded the older HTTP+SSE transport); the server runs as an HTTP service reachable by one or more clients; ideal for hosted/remote, multi-client servers. **Decision rule:** local tooling/desktop host → stdio; networked/multi-client/remote → Streamable HTTP. For HTTP, import `StreamableHTTPServerTransport` from `@modelcontextprotocol/sdk/server/streamableHttp.js`, create a per-session transport, and mount it on an endpoint (e.g. an Express `POST /mcp` piping `req`/`res` into `transport.handleRequest(...)`) — the tool definitions stay identical, only the transport swaps.

### 4. Connect it to an agent as a client
**(a) Smoke test — MCP Inspector:**
```bash
npx @modelcontextprotocol/inspector node build/server.js
```
List tools, call `add` with `a: 2, b: 3`, confirm `5`.

**(b) Programmatic client** — what a shipping agent does under the hood:
```ts
// client.ts
import { Client } from "@modelcontextprotocol/sdk/client/index.js";
import { StdioClientTransport } from "@modelcontextprotocol/sdk/client/stdio.js";

const transport = new StdioClientTransport({ command: "node", args: ["build/server.js"] });
const client = new Client({ name: "calc-client", version: "1.0.0" });

await client.connect(transport);
const { tools } = await client.listTools();          // what an agent feeds its LLM
console.log("Tools:", tools.map((t) => t.name));
const result = await client.callTool({ name: "add", arguments: { a: 2, b: 3 } });
console.log("add(2, 3) =>", result.content);
await client.close();
```
An LLM-driven agent replaces the two calls with a loop: pass `tools` to the model, let it choose `add` and its arguments, call `client.callTool`, feed the result back, repeat. Desktop hosts instead register the server via JSON: `{ "mcpServers": { "calc-server": { "command": "node", "args": ["/absolute/path/build/server.js"] } } }`. Expected client output: `Tools: [ 'add' ]` then `add(2, 3) => [ { type: 'text', text: '5' } ]`. To go remote, swap `StdioServerTransport` for `StreamableHTTPServerTransport` and add auth before exposing it.

## Efficient MCP server development practices

All guidance below targets the current spec revision, **2025-11-25** ([versioning](https://modelcontextprotocol.io/specification/versioning)).

**Tool & resource design.** Model the primitive to match *who drives the interaction*. **Tools** are model-controlled — the LLM discovers and invokes them autonomously — so use one for anything with a side effect or that the model should *decide* to call (query an API, run a computation, write a record). **Resources** are application-controlled readable data by URI — use one for addressable context (a file, a record, a document); return `resource_link` items from tools when a result points at large data rather than inlining it. **Prompts** are user-triggered templates ("summarize this PR"). Keep tools **narrow and composable** — one clear job each — so the model can chain them predictably. Prefer **idempotent** operations and signal safety via the `readOnlyHint` / `destructiveHint` / `idempotentHint` annotations (clients MUST treat these as untrusted hints). Return **structured, token-efficient** results: provide an `outputSchema`, populate `structuredContent`, and **paginate** large lists with cursors. Avoid a "god tool" with a `mode` parameter that does ten unrelated things, and avoid dumping multi-KB payloads into `content`.

**Schema clarity.** `inputSchema` is JSON Schema (defaults to 2020-12) and is the model's primary guide to *how* to call you; descriptions are its **only** guide to *when*. Mark `required` fields explicitly; use `enum` where the value space is fixed; use tight types (`integer` with `minimum`/`maximum`, `format: "date"`, patterns); and give every property a field-level `description`. Tool names use the charset `A–Z a–z 0–9 _ - .` (1–128 chars) with a human-readable `title` and an action-oriented `description` ("Create a calendar event. Returns the event ID."). **Validate server-side regardless of schema.**
```ts
server.registerTool("create_event", {
  title: "Create Calendar Event",
  description: "Create an event on the user's primary calendar. Returns the event ID.",
  inputSchema: {
    title: z.string().min(1).describe("Event title"),
    startIso: z.string().describe("Start time, ISO-8601 with timezone"),
    visibility: z.enum(["default","public","private"]).describe("Event visibility"),
  },
}, async ({ title, startIso, visibility }) => ({ content: [{ type: "text", text: "Created event …" }] }));
```

**Error handling — two distinct mechanisms.** **Protocol errors** → JSON-RPC `error` (unknown tool `-32602`, malformed request); recovery is unlikely, so reserve these for structural problems. **Tool-execution errors** → a normal `result` with `isError: true` and an actionable message in `content`, which clients SHOULD feed back to the model for self-correction. Return recoverable failures (bad date, upstream 429) as `isError` results with specific, actionable messages ("Invalid departure date: must be in the future. Current date is 2026-07-04."). Enforce timeouts on downstream calls; **never crash the process on bad input** — catch, classify, respond; and never leak stack traces, secrets, internal hostnames, or SQL into error text.

**Transport selection.** The spec defines exactly two standard transports — **stdio** and **Streamable HTTP** — and Streamable HTTP **replaces the deprecated HTTP+SSE transport** from protocol version 2024-11-05. Use stdio for local subprocess integration (lowest latency, credentials from the environment). Use Streamable HTTP for remote/multi-client servers that need OAuth 2.1, sessions, streaming notifications, and resumability. For HTTP servers: expose a single MCP endpoint supporting POST and GET; **validate the `Origin` header (403 on mismatch)** to prevent DNS-rebinding; **bind to `127.0.0.1`** when running locally, not `0.0.0.0`; and use cryptographically secure `MCP-Session-Id` values.

**Testing.** **Unit-test tool handlers in isolation** — call the handler directly with fixture inputs and assert on the returned `content` / `structuredContent` / `isError`; no transport needed. **Contract-test against your schemas** (generate invalid inputs, assert rejection; validate outputs against `outputSchema`). Use the official **MCP Inspector** (`npx @modelcontextprotocol/inspector node path/to/server.js`) for interactive Tools/Resources/Prompts inspection, capability negotiation, and a notifications pane. **Integration-test with a real client/agent** to confirm the model actually selects and calls your tools from their descriptions. Use deterministic fixtures — stub upstream APIs and inject "now" for date logic.

**Security.** Authorization is OPTIONAL in MCP, but remote servers inherit real obligations. Validate and sanitize every input server-side (MUST). For HTTP servers, act as an **OAuth 2.1 resource server**: advertise auth servers via Protected Resource Metadata (RFC 9728), require `Bearer` tokens on every request, enforce PKCE (`S256`), and use short-lived tokens. **Validate token audience** (RFC 8707 — accept only tokens issued specifically for your server), and **never do token passthrough** (forwarding the client's token to an upstream is forbidden and causes confused-deputy vulnerabilities). stdio servers should retrieve credentials from the environment rather than implement OAuth. Apply least privilege, request minimal scopes, rate-limit tool invocations, and treat tool outputs as untrusted content that can carry prompt injection into the agent.

**Performance.** Minimize token footprint (structured data plus `resource_link`s the client fetches on demand). Cache expensive lookups and reuse connections (DB pools, HTTP keep-alive) across tool calls. Stream/paginate long results. Handle concurrency safely — a Streamable HTTP server MUST NOT broadcast one message across multiple streams. Design **coarse-grained-enough** tools to avoid chatty round-trips: if the agent always calls `getUser` → `getOrders` → `getInvoice`, consider one tool that returns the joined view, since each observe→act loop is a full model round-trip.

## Near-term future-prominence forecast

**This section is a prediction, not established fact** — a projection extrapolated from the 2025 adoption trajectory and tied to explicit assumptions.

**Assumptions:** (1) major vendors (OpenAI, Google, Microsoft, AWS) sustain and deepen MCP client support rather than retreating to proprietary interfaces; (2) no dominant competing protocol displaces MCP in the window; (3) registry/server growth continues broadly upward (not necessarily at 2025's +407% pace); (4) no severe security or fragmentation setback erodes trust; (5) tooling and DX (auth, discovery, SDKs) keep maturing; (6) spec governance stays coherent and backward-compatible.

**Projection (through ~2027–2028), each hedged.** *Adoption breadth:* if assumptions 1–3 hold, MCP is likely to broaden from early-adopter dev tools into mainstream enterprise platforms; direction is more reliable than magnitude. *Standardization vs. fragmentation:* with four rival platforms already committed, MCP is well-positioned to become the default agent-tool interface; the main risk is soft fragmentation via vendor-specific extensions and auth variants, not a rival protocol. *Tooling maturation:* consolidation is projected around authentication, discovery/trust, and registry curation. *Enterprise uptake:* likely to accelerate as managed/remote MCP offerings (Google, AWS) reduce hosting friction, though enterprises will probably gate adoption on security and governance tooling.

**Scenarios.** *Base (most likely):* MCP entrenches as a widely-supported standard with imperfect consolidation and some competing conventions coexisting. *Upside (plausible, less likely):* MCP becomes the de facto universal standard with near-ubiquitous support and a trusted registry. *Downside (lower probability, non-trivial):* momentum stalls via fragmentation, a security incident, or a competing standard, and MCP persists as one option among several. **Watch-signals:** whether the four majors move MCP from preview to deep GA; the registry entry-count trajectory past ~2,000; emergence of any competing agent-tool protocol; maturation of MCP auth/identity standards; and reported production-usage metrics.

## Useful domains, and whether MCP-plus-agents can automate entire human workloads

In each domain below, MCP servers expose an existing system of record as tools (actions), resources (readable data), and prompts, so an agent connects as a client without a bespoke per-system integration:

- **Software engineering** — repos, issue trackers, CI, docs; read code context and invoke build/test/PR actions through one interface.
- **Customer support** — ticketing, knowledge bases, order/CRM data; retrieve account state, draft/route responses.
- **Data analysis / BI** — warehouses, dashboards, query engines; resource-exposed schemas let the agent compose and run queries with checkable outputs.
- **DevOps / infrastructure** — cloud APIs, logs, monitoring; read telemetry, invoke deploy/rollback/scaling.
- **Sales / CRM ops** — update pipeline records, enrich leads, generate follow-ups.
- **Knowledge management / internal search** — wikis, drives, doc stores; ground answers in retrieved context.
- **Finance ops** — reconcile transactions, pull ledger/ERP data, prepare reports where auditability matters.
- **Healthcare admin** — scheduling, coding, record lookups against EHR/practice-management systems (administrative, not clinical).
- **Legal research** — query case/document repositories, assemble draft memos with citations for human review.
- **Marketing ops** — pull analytics, push campaign/content actions across platforms.

**The automation boundary.** Place every task on a spectrum from **augmentation** (agent proposes, human decides/executes) to **full automation** (agent decides and acts unsupervised). MCP-plus-agents shifts many tasks leftward-to-middle; it does not move whole jobs to the right end. **Automatable now:** deterministic transforms with one correct form; verifiable outputs with a cheap success signal (tests pass, totals reconcile, schema validates); low-stakes/reversible actions (drafts, tagging, routing); high-frequency, well-scoped retrieval. **Not automatable now:** high-stakes, irreversible actions (moving money, deleting records, clinical/legal commitments); tasks needing accountable human judgment; ambiguous, open-ended goals with no checkable "done"; long-horizon multi-step chains where errors compound; tasks needing tacit knowledge absent from any system of record. The falsifiable core: if each autonomous step succeeds with probability *r*, an *N*-step chain with no correction succeeds roughly *r^N* — at r = 0.95, a 20-step task ≈ 0.36. A task class crosses into "automatable" when a measurable property changes: higher per-step reliability *r*, shorter effective *N* via human-in-the-loop checkpoints, cheap automated verifiers/guardrails that catch and retry, or making stakes reversible (sandboxes, dry-runs, approvals). **Bottom line:** MCP-plus-agents automates task *classes* and *portions* of workflows, not entire human jobs wholesale.

## Bringing it together: a builder's roadmap

If you are new to MCP and intend to ship, the ten sections above collapse into a single coherent path. This closing roadmap sequences them into decisions you make in order, so that each choice is informed by the one before it.

**Step 0 — Internalize the mental model, because it dictates every later choice.** MCP is a *specification*, not a product; a *server* provides tools/resources/prompts; your *agent (or its host) is the client* that consumes them. The reason this matters practically is that the same three primitives recur at every layer of the stack you are about to build. When you design a server, you are deciding which capabilities become model-controlled *tools*, which become application-controlled *resources*, and which become user-triggered *prompts*. When you choose a client, you are asking which of those primitives it can actually consume. And when you reason about the ecosystem's momentum, the load-bearing fact is the same one: a standard interface lets any client talk to any server, which is why M×N collapses to M+N and why the whole thing accelerated in 2025. Hold that one idea and the rest of the guide is just elaboration.

**Step 1 — Decide *why* you are adopting MCP, and confirm the payoff is real for you.** The M+N argument is the substance behind the 2025 buzz, and it is worth checking against your own situation. If you will only ever connect one agent framework to one tool, a hand-written adapter is fine and MCP's indirection buys you little. The protocol pays off precisely when you have *fan-out*: several agent applications (Claude Desktop, Cursor, VS Code Copilot, an OpenAI Agents SDK service) that all need to reach the same capability, or one agent that must reach many capabilities. In that shape, writing a server once and having it work across every compliant client is a large, compounding saving. The vendor commitments of 2025 — OpenAI (2025-03-26), Google (I/O 2025-05), Microsoft/GitHub (VS Code GA 2025-07-14), AWS (2025-10/11) — matter to you as a builder because they guarantee that the client you target next year is likely to speak the same protocol as the one you target today.

**Step 2 — Build the smallest possible working loop before anything else.** Do the getting-started runbook literally: `npm install @modelcontextprotocol/sdk zod`, paste `server.ts` and `client.ts`, `npx tsc`, then `npx @modelcontextprotocol/inspector node build/server.js` and watch your `add` tool return `5`. That single loop — server exposes a tool → Inspector lists and calls it → a programmatic client lists and calls it exactly as an agent would — *is* the whole protocol in miniature. Everything else is scaling this shape out: more tools, more resources, a remote transport, real auth. Getting this loop green first means every later problem is isolated to one added dimension rather than tangled with basic connectivity.

**Step 3 — Choose your transport from the deployment shape, not from preference.** The decision rule is mechanical: local tooling or a desktop host that launches your server as a subprocess → **stdio** (zero network setup, credentials from the environment); a networked, multi-client, or remotely-deployed server → **Streamable HTTP** (which superseded the deprecated HTTP+SSE transport). This choice is not cosmetic — it changes your security obligations entirely. A stdio server should *not* implement OAuth; a Streamable HTTP server exposed on a network inherits the full OAuth 2.1 resource-server checklist, must validate the `Origin` header, and should bind to localhost when local. Pick the transport that matches where the server will actually run, then adopt exactly the obligations that transport implies.

**Step 4 — Model your capabilities against the primitive semantics.** This is where good MCP servers separate from bad ones. A side-effecting action the model should decide to invoke is a *tool*; addressable context the host wants to attach is a *resource*; a reusable user-triggered workflow is a *prompt*. Keep tools narrow and composable, give every input field a tight type and a real description (the description is the model's only guide to *when* to call), signal safety with the `readOnly`/`destructive`/`idempotent` hints, and return structured, token-efficient results with `resource_link`s for bulk data. Get the error model right too: recoverable failures come back as `isError: true` results the model can self-correct on, while structural failures use JSON-RPC errors. These are the design decisions that determine whether an agent can actually chain your tools reliably.

**Step 5 — Pick clients and servers from the mature, convergent cluster.** You do not have to invent your ecosystem. On the server side, the browser-automation cluster (Playwright, Chrome DevTools, Browser Use, Puppeteer), the docs/reasoning helpers (Context7, Sequential Thinking), and Anthropic's seven reference servers (Filesystem, Fetch, Git, Memory, and the rest) recur across independent directories — a safe starting palette. On the client side, the breadth leader is the IDE/coding-agent tier (VS Code Copilot, Cursor, Claude Code, Cline, Continue), backed by programmatic hosts (OpenAI Agents SDK, LangChain/LangGraph, Goose, Copilot Studio). Match the client to the *primitives* you rely on: if your server leans on resources or prompts, confirm the target client actually consumes them, because "supports MCP" varies by primitive.

**Step 6 — Set expectations for scope and future, honestly.** MCP-plus-agents automates task *classes* and *portions* of workflows — deterministic, verifiable, reversible, well-scoped work — not entire jobs, because unverified multi-step chains fail by compounding (r^N). Design for the boundary: insert human-in-the-loop checkpoints, add cheap verifiers, and keep destructive capability scoped so a hijacked prompt cannot reach it. As for prominence, the reasonable planning stance is the base-case projection: MCP is well-positioned to consolidate as the default agent-tool interface through 2027–2028 under the stated assumptions, with soft fragmentation (vendor extensions, auth variants) as the likelier risk than outright displacement. Build as though the standard endures, but keep your server's business logic cleanly behind the MCP boundary so that, if the wire protocol shifts, you re-wrap rather than rewrite.

Follow those six steps in order and you will have used every part of this guide — definitions, the M+N rationale, the build loop, transport and primitive design, the ecosystem map, and a calibrated view of scope and future — as a single connected path from "new to MCP" to "shipping working software."

## Verification notes

- **Confidence on the core premise.** The reading that 2025 MCP adoption was genuine and measurable (not marketing) is **high-confidence**; the strongest evidence is dated competitor vendor commitments, not counts. Some vendor support first shipped as preview/experimental before GA — treat "adoption" as a spectrum, not a binary.
- **Download and count figures are mostly 2026 retrievals.** The large SDK download volumes (167.5M npm/month, ~1.46B PyPI total) were retrieved in ~July 2026, not clean 2025 series; they demonstrate sustained scale but were deliberately kept *out* of the load-bearing 2025 claims. Headline "server count" figures in the tens of thousands are catalog-dependent, double-count servers listed in multiple directories, and include duplicate/abandoned entries; catalogs disagree (e.g., Glama reported 50,845 vs. ~37,000 across sources). Stars and downloads are not production usage, and package downloads include CI/bot traffic.
- **"Most used" servers are proxied, not censused.** There is no authoritative global usage census. PulseMCP "visitors/30-day" measures directory traffic (favoring browser-automation servers); mcp.directory "installs" are far smaller absolute numbers (hundreds); and GitHub stars often belong to a *parent project* (Browser Use ~102k, n8n ~194k) rather than the MCP server itself. Sources agree on the leading cluster but disagree on ordering and scale (e.g., Context7 is #1 on mcp.directory but #4 by PulseMCP traffic). Read the roster as directional, not a precise leaderboard.
- **Server-provenance corrections.** The "GitHub" MCP server is now officially GitHub-maintained (api.githubcopilot.com/mcp), despite some directory labels attributing it to Anthropic. Anthropic's earlier Slack, Google Drive, Google Maps, PostgreSQL, Redis, and Brave Search integrations are *archived* in the servers repo — historically prominent but no longer first-party-maintained.
- **Client matrix caveats.** The example-clients matrix is a *living, self-reported* document (apps add themselves), so inclusion proves capability, not popularity; no per-app active-user counts are publicly available. **JetBrains and Replit** are widely claimed as MCP hosts but had no first-party client-support page confirmed in this pass — treat as "claimed, evidence not found."
- **Illustrative numbers.** The compounding-error figures (r = 0.95, 0.95²⁰ ≈ 0.36) and the "~15–20 unverified steps" threshold are pedagogical/testable hypotheses, not measured from a cited source; no per-step reliability dataset underlies them.
- **Forecast.** Section "Near-term future-prominence forecast" is a prediction, not fact — moderate confidence for the base case and low confidence on any specific magnitude or milestone date, extrapolated from roughly one year of adoption data.
- **SDK/spec currency.** Spec claims target revision **2025-11-25**. `registerTool(name, config, handler)` with a Zod *raw shape*, `StdioServerTransport`/`StdioClientTransport`/`Client`, and `.js` import extensions under NodeNext ESM are the current documented TypeScript SDK symbols; older code uses the legacy `server.tool(...)` form. Verify exact SDK option keys against your installed version, since helper APIs evolve faster than the wire spec. No specific SDK version numbers are asserted. The Streamable HTTP handler is described rather than fully coded; follow the `streamableHttp` example in the SDK repo.
